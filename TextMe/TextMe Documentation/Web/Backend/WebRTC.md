# 📡 WebRTC Deep Dive: Как работают звонки в TextMe

В этом документе мы разберем «магию» реального времени под капотом нашего приложения. Мы пройдем путь от сигнального сервера до установления прямого зашифрованного канала между браузерами.

---

## 🏗️ 1. Архитектура Сигнализации (Signaling)

WebRTC позволяет браузерам обмениваться данными напрямую (P2P), но сначала им нужно «увидеть» друг друга. Для этого нужен посредник — **Signaling Server**. В нашем случае это **SignalR** на базе ASP.NET Core.

### 🤝 Тройное рукопожатие:
1.  **Offer (Оффер)**: Инициатор создает SDP (описание сессии) и отправляет его через сервер.
2.  **Answer (Ответ)**: Получатель принимает оффер и отправляет свой ответный SDP.
3.  **ICE Candidates**: Устройства обмениваются сетевыми маршрутами (как и куда слать данные).

---

## 🧱 2. Проблема NAT: Почему всё не так просто?

Большинство устройств в интернете находятся за роутерами. Роутер использует технологию **NAT (Network Address Translation)**.

> [!IMPORTANT]
> **NAT** — это как консьерж в многоквартирном доме. Весь мир видит адрес дома (публичный IP), но не знает номер вашей квартиры (локальный IP). Из-за этого браузер собеседника не может просто «постучаться» к вам напрямую.

---

## 🪟 3. Решение: STUN-сервер («Зеркало»)

Чтобы пробиться через NAT, браузеру нужно узнать свой публичный IP-адрес и порт, который открыл роутер.

**STUN (Session Traversal Utilities for NAT)** работает как зеркало:
1.  Браузер шлет запрос на STUN-сервер (Google).
2.  Сервер отвечает: *"Привет, я вижу тебя по адресу 95.1.2.3 с порта 54321"*.
3.  Теперь браузер может передать этот адрес собеседнику.

**Код конфигурации:**
```typescript
const iceServers = {
    iceServers: [
        { urls: "stun:stun.l.google.com:19302" } // Бесплатный сервер от Google
    ]
};
```

## 📦 4. ICE Candidates: Сбор маршрутов

**ICE (Interactive Connectivity Establishment)** — это процесс «прощупывания» всех возможных путей соединения.

### Типы кандидатов:
-   **Host**: Прямое соединение в одной Wi-Fi сети (самое быстрое).
-   **Srflx (Server Reflexive)**: Публичный IP, полученный через STUN (основной способ в интернете).
-   **Relay**: Трафик через TURN-сервер (если ничего другое не сработало).

**Обмен кандидатами (Фрагмент `useWebRTC.ts`):**
```typescript
peer.onicecandidate = async (e) => {
    if (e.candidate) {
        // Каждый найденный путь мы немедленно шлем собеседнику через SignalR
        await (chatHub as any).connection?.invoke("SendIceCandidate", targetUserId, e.candidate);
    }
};
```

---

## 🔀 5. TURN-сервер: Последний рубеж

Иногда роутеры очень «строгие» (например, в корпоративных сетях). Они блокируют прямые соединения даже при наличии STUN.

> [!NOTE]
> Если STUN не сработал, WebRTC использует **TURN (Traversal Using Relays around NAT)**. 
> Это сервер-посредник, который просто перекачивает трафик через себя. Это работает в 100% случаев, но создает нагрузку на сервер. (В текущей версии TextMe мы полагаемся на STUN).

## 🎙️ 6. Передача и анализ потока

### Скрытие микрофона (Mute)
Мы управляем `enabled` свойством аудио-дорожки. Это позволяет мгновенно отключать звук без разрыва соединения.

**Код реализации:**
```typescript
const toggleMute = () => {
    if (localStream) {
        const audioTrack = localStream.getAudioTracks()[0];
        // Выключаем саму дорожку, браузер перестает слать данные
        audioTrack.enabled = !audioTrack.enabled;
        setIsMuted(!audioTrack.enabled);
    }
};
```

### Voice Activity Detection (VAD)
Приложение анализирует входящий аудио-поток для визуальных эффектов.

**Как это работает (Web Audio API):**
```typescript
// Анализируем громкость входящего потока
const dataArray = new Uint8Array(analyser.frequencyBinCount);
analyser.getByteFrequencyData(dataArray);

// Считаем среднюю громкость
const average = dataArray.reduce((p, c) => p + c, 0) / dataArray.length;

// Если громкость выше порога — считаем, что человек говорит
if (average > 15) {
    setIsRemoteTalking(true);
} else {
    setIsRemoteTalking(false);
}
```

## 🚀 7. Жизненный путь звонка (P2P vs Server)

| Этап | Роль Сервера (Render) | Роль Браузера |
| :--- | :--- | :--- |
| **Вызов** | Передает Offer | Создает SDP, захватывает камеру |
| **Сетевой поиск** | Передает ICE Candidates | Опрашивает STUN-серверы |
| **Соединение** | **Отключается от процесса** | Устанавливает прямой P2P туннель |
| **Разговор** | 0% участия (спит) | 100% трафика (шифрование и передача) |
