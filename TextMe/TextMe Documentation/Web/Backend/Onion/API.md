# Документация Инфраструктурного Слоя в TextMe (Infrastructure)

Слой **API** является точкой входа в приложение (Host). Это "лицо" бэкенда, которое принимает HTTP-запросы от фронтенда и мобильного приложения, обрабатывает их и возвращает результат.

## Структура папок

### 1. `Controllers/`
Классы-контроллеры, определяющие маршруты (Routes) API. Каждый метод здесь соответствует определенному действию: `GET /api/messages`, `POST /api/auth/login` и т.д.
*   **Для чего:** Группировка эндпоинтов по смыслу. Контроллеры не содержат логики — они лишь вызывают нужные сервисы из слоя Application.

### 2. `Middlewares/`
Промежуточное ПО, которое перехватывает запросы до того, как они попадут в контроллер.
*   **Пример:** `ExceptionMiddleware` ловит все ошибки в приложении и возвращает клиенту красивый JSON с описанием ошибки вместо технического "экрана смерти".
*   **Для чего:** Глобальная обработка ошибок, логирование и проверка безопасности.

### 3. `Extensions/`
Статические методы расширения для настройки проекта. Здесь настраиваются:
*   **CORS**: Разрешение доступа для фронтенда на Netlify и мобильного приложения.
*   **Swagger**: Автоматическая генерация документации API.
*   **Dependency Injection (DI)**: Регистрация всех сервисов и репозиториев.
*   **Для чего:** Позволяет держать файл `Program.cs` чистым и читаемым.

### 4. `Properties/` & `appsettings.json`
Конфигурационные файлы.
*   `appsettings.json`: Хранит настройки подключения к БД, секретные ключи JWT, настройки Cloudinary.
*   **Для чего:** Позволяет менять поведение приложения без пересборки кода.

### 5. `Program.cs`
Главный файл запуска приложения. Здесь инициализируется веб-сервер, подключаются все расширения (Extensions) и настраивается конвейер (Pipeline) обработки запросов.

## Почему это хорошо?

1.  **Чистый вход**: Благодаря выносу настроек в `Extensions`, основной файл запуска остается очень лаконичным, и сразу видно, какие технологии используются в проекте.
2.  **Безопасность**: Слой API отвечает за первичную валидацию токенов и CORS, не пропуская вредоносные запросы к бизнес-логике.
3.  **Документированность**: Встроенный Swagger позволяет фронтенд-разработчикам сразу видеть список всех доступных методов, их параметры и примеры ответов.
4.  **Тонкие контроллеры**: Мы следуем правилу "Skinny Controllers", что делает код легким для чтения и поддержки. Вся сложная работа происходит "под капотом" в других слоях.
