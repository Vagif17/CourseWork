# Документация Прикладного Слоя в TextMe (Application)

Прикладной слой является «мозгом» системы. Здесь сосредоточена вся бизнес-логика и правила работы мессенджера. Этот слой координирует выполнение задач, не заботясь о том, как данные хранятся или как они передаются пользователю.

---

## 1. Интерфейсы и Абстракции (Interfaces)
Слой Application определяет контракты, которые должны реализовать другие слои. Это обеспечивает гибкость и независимость кода.

- **IRepositories**: Интерфейсы для доступа к данным (`IChatRepository`, `IMessageRepository`).
- **IServices**: Контракты для бизнес-сервисов (`IAuthService`, `IUserService`).
- **INotifications**: Интерфейсы для рассылки уведомлений в реальном времени.
- **IStorages**: Абстракции для работы с облачными хранилищами файлов.

---

## 2. Модели передачи данных (DTO'S)
**DTO (Data Transfer Objects)** — это простые объекты, которые определяют структуру данных для входящих и исходящих запросов API.

- **Request DTOs**: Модели для приема данных от клиента (`LoginRequest`, `CreateChatRequest`).
- **Response DTOs**: Модели для формирования ответов клиенту (`MessageResponse`, `ChatDto`).
- **Изоляция**: Использование DTO гарантирует, что внутренние сущности базы данных (Domain) никогда не попадут напрямую к клиенту.

---

## 3. CQRS и Функциональные блоки (Features)
Для масштабируемости используется паттерн **CQRS** (Command Query Responsibility Segregation). Логика разделена на изменение данных и их чтение.

- **Commands**: Действия, меняющие состояние системы (`CreateMessageCommand`, `EditMessageCommand`).
- **Queries**: Запросы на получение данных без побочных эффектов (`GetChatByIdQuery`).
- **Handlers**: Классы-обработчики, которые содержат саму логику выполнения конкретной команды или запроса (могут находиться в отдельной папке или рядом с командами).

---

## 4. Валидация и Маппинг (Validators & Mapping)
- **FluentValidation**: Каждый входящий запрос проходит строгую проверку. Например, `UserRegisterValidator` следит, чтобы почта была введена корректно, а пароль соответствовал требованиям безопасности.
- **AutoMapper Profiles**: Наборы правил для автоматического преобразования данных между сущностями Domain и DTO. Это избавляет от написания тысяч строк кода типа `a = b`.

---

## 5. Бизнес-сервисы (Services)
Высокоуровневая логика, которая оркестрирует работу нескольких компонентов.

- **AuthService**: Логика входа, регистрации и управления сессиями.
- **ChatService**: Управление участниками групп и правами доступа.
- **NewsService**: Логика получения и обработки новостей перед отправкой пользователю.

---

## Особенности реализации

### Безопасность данных
Application слой гарантирует, что бизнес-правила соблюдаются всегда. Например, при попытке удалить сообщение проверяется, является ли текущий пользователь его автором.

### Конфигурация (Config)
Используется паттерн **Options**. Все настройки из `appsettings.json` маппятся на строго типизированные классы (`JwtConfig`, `CloudinaryConfig`), что исключает ошибки при чтении настроек.
